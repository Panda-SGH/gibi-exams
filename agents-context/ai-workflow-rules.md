# AI Workflow Rules

## Overall Workflow

* Work in a spec-driven manner at all times.
* Implement only what is explicitly defined in the current task scope.
* Build the project incrementally in small, verifiable units.
* Complete one feature unit before starting another.
* Treat architecture documents, product requirements, and workflow documents as the source of truth.
* Preserve existing architecture boundaries during implementation.
* Prefer simple, maintainable solutions over clever abstractions.

---

## Scope Control Rules

* Work on one feature unit at a time.
* Do not implement multiple unrelated systems in a single change.
* Do not perform speculative refactors.
* Do not introduce features, abstractions, dependencies, or infrastructure that are not required by the current task.
* Do not “prepare for future features” unless explicitly instructed.
* Do not modify unrelated files during implementation.
* Keep pull requests and commits focused on a single responsibility.
* If a task becomes too large, split it into smaller implementation units before continuing.

---

## Task Decomposition Rules

* Split work into smaller steps whenever a task:

  * touches multiple architectural boundaries
  * requires database and UI changes together
  * introduces asynchronous/background behavior
  * affects authentication or authorization
  * modifies shared infrastructure
  * cannot be safely verified in a single pass

* Break large tasks into:

  1. schema/infrastructure
  2. backend logic
  3. API/server actions
  4. UI integration
  5. validation and testing

* Finish and verify each layer before moving to the next.

* Do not build UI for backend systems that do not exist yet.

* Do not connect frontend components to incomplete backend implementations.

---

## Requirements Handling Rules

* Never guess missing requirements.

* Never invent product behavior.

* Never assume undefined edge-case behavior.

* Stop implementation immediately when requirements become ambiguous.

* Explicitly surface missing decisions or unclear behavior.

* Request clarification before continuing if:

  * data ownership is unclear
  * validation behavior is undefined
  * permissions are unspecified
  * UI interactions are ambiguous
  * failure states are not defined
  * storage behavior is unclear

* If blocked by ambiguity:

  * document the blocker clearly
  * isolate completed work
  * avoid speculative implementation

---

## Architecture Enforcement Rules

* Keep business logic outside UI components.
* Access the database only through the data access layer.
* Perform mutations only through server actions or API routes.
* Keep Telegram bot logic isolated inside `lib/bot`.
* Keep webhook handlers thin.
* Keep client components presentation-focused.
* Never expose secrets or sensitive logic to the client.
* Keep storage access controlled through backend functions only.
* Preserve folder ownership boundaries defined in `architecture.md`.

---

## File Modification Restrictions

Do not modify the following without explicit instruction:

* components/ui/*`
* generated shadcn components
* autogenerated migration snapshots
* lockfiles unless dependency changes are required
* environment configuration templates
* deployment configuration
* generated types
* formatting/lint configuration
* CI/CD configuration
* database migration history already applied in production

Additional rules:

* Do not rename folders or move architectural boundaries without approval.
* Do not replace foundational libraries without explicit instruction.
* Do not rewrite stable working systems during feature implementation.

---

## Documentation Synchronization Rules

* Keep documentation synchronized with implementation at all times.
* Update documentation immediately after architectural or behavioral changes.
* Reflect database schema changes in documentation.
* Reflect API contract changes in documentation.
* Reflect workflow or infrastructure changes in documentation.
* Do not leave TODO documentation placeholders for completed systems.
* Remove outdated documentation during refactors.
* Ensure examples remain accurate after implementation changes.

Documents that must stay synchronized include:

* `architecture.md`
* `ai-workflow-rules.md`
* setup documentation
* environment variable documentation
* database schema documentation
* API usage documentation

---

## Verification Rules

Before marking a task complete:

* Verify the implementation matches the specification exactly.
* Verify no unrelated behavior changed.
* Verify architectural boundaries remain intact.
* Verify TypeScript types are correct and complete.
* Verify loading, empty, and error states exist.
* Verify authentication and authorization rules are enforced server-side.
* Verify no secrets are exposed to the client.
* Verify mobile responsiveness for user-facing UI.
* Verify imports and dependencies are minimal and necessary.
* Verify dead code and unused files are removed.
* Verify linting and formatting pass.
* Verify documentation has been updated.
* Verify the feature works from the real user flow perspective.

---

## Quality Enforcement Rules

* Extract repeated logic once duplication appears more than twice.
* Keep functions small and single-purpose.
* Prefer explicit code over hidden magic behavior.
* Use descriptive naming consistently across frontend, backend, and database layers.
* Document complex reasoning with concise comments explaining why.
* Avoid deep component nesting.
* Avoid premature optimization.
* Avoid overengineering.

---

## Safety Rules

* Never bypass validation layers.
* Never trust client-provided metadata.
* Always validate uploaded files before publishing.
* Always verify Telegram authentication server-side.
* Never disable security protections for convenience.
* Never weaken RLS policies without explicit instruction.
* Never use unsafe SQL string interpolation.
* Never commit secrets, tokens, or credentials.

---

## Completion Rules

A feature unit is complete only when:

* implementation is functional
* verification passes
* documentation is updated
* architecture rules remain intact
* no speculative code remains
* no placeholder logic remains
* no unrelated files were modified
* the feature can be understood by another developer without additional explanation
